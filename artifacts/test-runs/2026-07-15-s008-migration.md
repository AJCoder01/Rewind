## Applied behavior

- The migration runner compared the migration bytes with the reviewed checksum before opening a connection.
- The first `npm run db:migrate` applied the migration atomically and recorded its checksum.
- A second `npm run db:migrate` recognized the exact ledger entry, revalidated the complete catalog, and did not execute the migration again.
- The runner used a deterministic search path, bounded timeouts, one transaction-scoped advisory lock, ledger validation, and rollback on failure.
- The applied migration is now immutable. Any future schema change must use a new numbered migration.

## Live database proof

`npm run db:verify` passed every reported check:

- restricted runtime identity, TLS, schema access, and role flags;
- migration-owner identity and TLS;
- exactly 11 Rewind tables (one technical ledger plus ten application tables);
- exact owners, columns, nullability, defaults, sequence, and all 26 named constraints;
- all required primary-key, unique, check, and foreign-key definitions;
- exact runtime table/sequence grants and read-only ledger access;
- no effective table/sequence access for `PUBLIC`, `anon`, `authenticated`, or `service_role`;
- exact migration-owner default ACLs;
- exactly one matching migration-ledger row;
- rolled-back `(plan_id, action_key)` and idempotency uniqueness probes;
- rolled-back check-constraint and foreign-key probes, with no residue;
- live database readiness; and
- rejection of a plaintext non-local connection for an SSL-specific reason.

An initial readiness run exposed a driver-boundary defect: an array of PostgreSQL `name` values was returned as a serialized string by the pooler. The readiness query now casts `table_name` to `text`; a regression assertion was added before the final successful live verification.

The hardened ACL verification also identified the PostgreSQL global default `PUBLIC EXECUTE` function privilege that cannot be removed by a schema-scoped revoke. The migration-owner connection applied the owner-level global revoke, and the final verifier confirmed both global and `public` function defaults are owner-only.
